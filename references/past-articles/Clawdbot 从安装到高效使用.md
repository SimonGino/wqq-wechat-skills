你以为你缺的是一个更聪明的对话框。  
真正缺的往往是：**一个能把事情做完的“同事”**。

大多数 AI 帮你写得很漂亮，但最后那 20 分钟还是你自己花掉：开网页、找资料、改格式、发消息、建日程、把文件放进正确的文件夹……

它把你日常的聊天软件（Telegram/WhatsApp/Discord/Slack/iMessage…）变成“控制台”：你在聊天窗口交代目标，它在你的设备上跑流程，最后把交付物扔回来。

## 一分钟讲清：Clawdbot 是什么（以及它不是什么）

一句话定义：**Clawdbot 是开源、自托管的 AI Agent**。  
你可以把它理解成“听得懂自然语言的系统管理员”：

- 大模型负责“想”：推理、拆解任务、做决策
- 你的设备负责“做”：读写文件、跑命令、自动化网页、处理工作流（能力取决于你的配置与权限边界）
- 聊天软件负责“交互”：你发指令，它回交付物

所以它**不是**“又一个聊天机器人”，也不是“更会写的 ChatGPT”。  
它更像一套系统：把 AI 的大脑接到你机器的手脚上，长期运行，长期沉淀。

外网有个很常见的“黑话”解释，也挺形象：

- Clawd：通常指 Claude/Claude Code（带点调侃意味，“效率怪物化”的 Claude）
- Clawdbot：可以理解成 “Claude Code + Bot 化工作流”，把 Claude 的能力塞进一个常驻的执行体里

---

## 为什么要用它：你买到的是“执行力”，不是“聪明”

如果你只是想问答、写文案、润色句子，任何大模型都够用。  
Clawdbot 的价值点在于：它会吞掉你最烦的那类碎活——跨应用、重复、流程长、但又不得不做。

典型痛点：

- 你每天在 8 个 App 里切来切去：聊天、日历、邮箱、浏览器、文件夹、笔记、工单、表格
- 一件事拆成很多步：查资料→整理→写→发给人→同步到待办→跟进
- 你不想把日程/文件/账号权限交给一堆 SaaS：但又想要自动化

Clawdbot 的关键收益常见是三类：

- ChatOps：把流程收敛到一个窗口，“一句话→交付物”
- 自托管：数据与工作流尽量留在你可控的环境里
- 可复用：重复流程能沉淀成 Skill（技能），越用越省心

除此之外，还有两个很“上头”的点，经常是大家觉得它“牛逼”的原因：

- 长期记忆：你反复提到的偏好、上下文、工作方式，会逐渐被固化到工作区/记忆里，越用越像你
- 主动联系：任务完成、需要你确认、或者触发了提醒/监控，它可以主动给你发消息，而不是等你来问

## 先把账算清：你要付出什么成本

这类工具最怕“只讲好处不讲代价”，先把现实摆在台面上：

- 一次性成本：第一次安装与接渠道，通常 30-90 分钟能跑通“第一条消息”
- 持续成本：需要一台尽量在线的设备（本机、Mac mini、家里小主机、轻量 VPS 都行）
- 模型成本：云模型会有 API/套餐开销；本地模型用算力换成本（效果上也要接受权衡）

更具体一点的“常见账单结构”是：

- 服务器：用家里机器基本为 0；用云服务器通常是低配常驻（费用随地区/配置波动，也可能用免费额度先试跑）
- 模型：订阅/调用成本是主头；你用得越像“同事”，调用就越频繁，所以“稳定 + 抗用”比“最强”更重要

这类成本最好跟“人力时间”比：  
如果它每天替你吞掉 30 分钟碎活，一周就能明显感到回报；如果你把它当“聊天玩具”，就很难算得过来。

如果你完全不想折腾安装、权限、常驻运行，那自托管 Agent 这条路大概率不适合你。  
如果你愿意为“可控 + 可沉淀 + 可复用”付出一次搭建成本，回报会很快出现。

## 安装前条件：别急着跑命令，先看这张清单

最低门槛（建议一次到位）：

- macOS / Linux / Windows（Windows 推荐 WSL2）
- Node.js 22+（尽量 LTS）
- 一个模型来源（OpenAI/Anthropic API 或本地模型）

体验加分项（强烈建议）：

- 一台“专门跑它”的机器：Mac mini / 小主机 / NAS 旁边一台 Linux 盒子
- 远程访问用内网组网：Tailscale/WireGuard 等，把服务放在内网跑

---

## 部署怎么选：三种姿势，选对了后面会省很多事

Clawdbot 跑在哪里，决定了你后面“稳定性、权限边界、成本”三件事。

- 跑在你的工作机：好处是安装快、权限天然够用，适合先体验；风险是关机/合盖就停，且权限太大容易“一不小心放飞”
- 跑在专用设备：形态常见是 Mac mini/家里小主机/低功耗 Linux 盒子；好处是常驻在线、权限可控、体验稳定；Mac mini 这种低功耗又能打的设备因此在社区里很常见；建议用单独系统账号运行 + 目录沙箱
- 跑在 VPS：好处是随时在线、成本可控、部署灵活；注意别为了“像本地一样强”去堆配置；经验是先低配跑起来，稳定后再加，远程尽量走内网组网而不是暴露公网面板

## 20 分钟跑通路线：安装 → Onboard → 启动 Gateway → 在 IM 里回你一句

注意：不同版本的交互/命令可能略有差异，以安装脚本与向导提示为准。下面按最常见路线写。

### Onboard 里关键选项怎么选：少走弯路的“默认答案”

你第一次跑向导时，最容易纠结的其实不是“怎么装”，而是“怎么选”。我给一个偏稳的选择顺序：

- QuickStart：先选它。目标只有一个：跑通第一条消息。
- Model Provider：你已经在哪家付费/有额度，就选哪家。不要为了“最强”先折腾。

![image-20260125232916988](https://oss.mytest.cc/2026/01/c082d6d9d5d30f31de0a613819ded5b4.png)

- 鉴权方式：向导里一般会给多种方式（API key、setup token、CLI 登录等）。优先选你最稳定能用的那种。
- 渠道（Channel）：先选一个，不要一口气全接。

![image-20260125232857102](https://oss.mytest.cc/2026/01/a9271f18232a5192d623017a6f216475.png)

- Skills/Hooks：不熟就跳过。先把“能执行”的主干跑通，扩展能力是后面的事。

你会看到一个非常关键的步骤：**启动 Gateway**。  
它相当于“常驻执行中枢”，后面你接入多少平台、跑多少技能，本质都围绕它运转。

### 1）安装（macOS / Linux）

```bash
curl -fsSL https://clawd.bot/install.sh | bash
```

如果你更习惯 npm，也可以用 npm 全局安装（适合你想自己管理 Node 版本/全局包的情况）：

```bash
npm i -g clawdbot
```

装完后你会进入初始化向导。新手建议：

- 风险提示：建议认真看一遍并同意（它具备执行能力）
- Onboard 模式：选 QuickStart（先跑通再细调）
- Model Provider：按你已有账号选（Anthropic/OpenAI/本地模型）
- 渠道：先选一个（Telegram 或 WhatsApp 二选一就够了）
- Skills/Hooks：不熟先跳过

当你看到“启动 Gateway”并出现网页面板入口时，说明你已经跑通了一半。

### 2）安装（Windows）

```powershell
iwr -useb https://clawd.bot/install.ps1 | iex
```

Windows 更建议在 WSL2 里按 Linux 的方式跑，工具链兼容性更好。

### 3）装完先体检（强烈建议）

在正式投入使用前，建议运行项目自带的 `doctor` 命令做环境健康检查，并尽量只在受信任的网络环境（VPN/内网）暴露服务接口。

## 安装后你会在哪用：一个“控制台”，一个“工作台”

你会获得两个入口：

- 网页面板：偏管理、调试、看状态（把它当控制台）
- 聊天软件：偏日常执行与交付（把它当工作台）

真正的爽点来自聊天软件：你在 IM 里交代目标，它在后台执行并回传结果。

---

## 平台怎么接入：先选一个你最常用的入口

所有渠道的基本逻辑都一样：

- 平台侧：拿 token / 扫码登录
- Clawdbot 侧：完成绑定
- 首次对话：触发 Pairing（配对）  
配对没批准，它就不接单，这是安全底线之一

下面把两条最常见路线写细一点。

### Telegram：新手最推荐（稳、好排障）

1. 打开 Telegram，和 `@BotFather` 对话
1. 用 `/newbot` 创建机器人，拿到 token
1. 在 Clawdbot 向导/配置中填入 token
1. 在 Telegram 里和你的 bot 对话，拿到 pairing code
1. 在 Clawdbot 侧确认配对（看到“已批准/已配对”即可）

一个很关键但经常被忽略的小步骤：**把 Telegram 的 user ID 绑进 allowlist**。  
你可以通过 Telegram 的 user id 查询 bot 获取自己的 ID（常见的是 `@userinfobot` / `@userIdBot` 这类），再把 ID 填回向导/配置里。这样就能做到“只有你能指挥它”。

### WhatsApp：最像“随身助理”（随口一句话就能下单）

典型流程：

1. 在 Clawdbot 中选择 WhatsApp 渠道
1. 按提示完成登录（通常是手机端“关联设备”扫码）
1. 完成配对确认，限制只有你的账号能指挥它

其他渠道（Discord/Slack/Signal/iMessage…）你按同一逻辑迁移即可：  
先把接入打通，再把 pairing 与 allowlist 做好，再谈能力扩展。

## 第一次开机：5 分钟让它“活过来”（别跳过）

很多人以为“接入成功”就结束了，其实你还差最后一步：**给它身份、目的与时区**。  
第一次在 IM 里对话时，它通常会问你类似这些问题：

- 我应该怎么称呼你？
- 你希望怎么称呼我？
- 我的主要职责是什么？（建议写 3 条，越具体越好）
- 你在哪个时区？（这会直接影响提醒/日程类任务）

把这一步认真答完，后面它的行为会稳定很多：你是在“养一个长期同事”，不是一次性对话。

---

## 你的前三个“立刻见效”测试：5 分钟建立信任

你刚装好时最需要的是：**立刻看到它在干活**。下面三条是最通用的“验证题”：

**测试 1：信息筛选（无门槛）**  
指令：  
“我接下来 30 分钟最应该处理哪 3 件事？基于：待办/对话/我下面贴给你的内容，给我 3 条建议 + 1 个最小下一步动作。”

**测试 2：研究总结（无门槛）**  
指令：  
“研究一下【公司/产品/项目】做什么：给我 3 条一句话结论 + 3 个风险点 + 我下一步该问的问题清单。”

**测试 3：收件箱/提醒（取决于你是否已接入工具）**  
如果你已经接入邮箱/日历：  
“检查我最近 10 封邮件（或今天的日程）：哪些必须回？给我‘该回/可不回/可延后’三类清单，并帮我起草 3 封最关键的回复。”  
如果你还没接入邮箱/日历：  
你可以先把 5-10 封邮件内容（或截图/转发文本）丢给它，让它用同样的格式输出“该回/可不回/可延后”清单。

这三条跑通，你基本就会从“试试看”变成“开始交活”。

---

## 强烈建议：用语音下单（体验会直接翻倍）

你不需要把它当“打字机器人”。  
Telegram/WhatsApp 的语音消息是 Clawdbot 的黄金入口：你走路、通勤、做家务时把任务说出来，它负责整理成清单、拆解步骤、生成交付物。

有些人还会在 Slack/Discord 里开多个 agent/频道并行跑任务：一个做调研，一个写稿，一个跑工具链。你不必一开始就这么玩，但这就是它能“持续放大你产能”的原因。

---

## 远程怎么用更稳：别暴露公网，先把“路”修好

你真正会长期用上 Clawdbot 的场景，往往是“人不在机器旁边”。  
这时候最稳的思路不是开公网端口，而是把你的设备拉进一个虚拟局域网里：

- 内网组网：Tailscale/WireGuard 这类工具把手机、电脑、服务器放进同一个网段
- 远程桌面（可选）：需要可视化操作时再上 VNC/远程桌面

这样做的好处很直接：

- 安全边界更清晰：面板与服务不需要暴露公网
- 体验更稳定：你像在本地一样访问它

---

## 模型怎么选：别纠结“最强”，先追求“稳定 + 够用”

Agent 和纯聊天不一样：它要跑流程、要调工具、要长期用。  
所以模型选择我更建议按这个顺序：

1. 稳定（别动不动授权掉线、工具调用失败）
1. 成本可承受（你会比纯聊天用得更勤）
1. 能力匹配（重度编码/复杂规划才真的需要顶配）

一个很实用的做法是：日常任务用“更抗用的模型”，关键任务再切到“更强的模型”。  
你不用一次性把所有需求都押在同一个模型上。

---

## 让它真的“会查资料”：接入 Web Search（Brave Search 例）

很多人第一次觉得 Agent “像人”，是从它会自己查资料开始的：  
不是你把资料喂给它，而是它能主动检索、交叉验证、再把结论结构化输出。

最常见的做法是接一个 Web Search Provider（例如 Brave Search）。思路很简单：

- 去对应服务申请一个 API key（通常有免费额度/试用）
- 在 Clawdbot 的配置/向导里把 key 填进去
- 让它做一次“研究总结”任务，确认它真的能检索到网页信息

接入后，你可以把“研究总结”升级成更强的指令：

- “研究【公司/竞品】并给我一页纸：一句话结论、业务模型、优势劣势、近期动态、我下一步该问的问题。”

---

## 配置要点：你越想“自动化”，越要先把边界写死

Clawdbot 的价值来自“能执行”，风险也来自“能执行”。建议至少守住这些底线：

- Allowlist：只允许你自己的账号/ID 下指令
- 目录沙箱：文件读写限制在 workspace 之内（别一上来全盘可写）
- 最小权限：能不用 root 就不用；能只读就别给写
- 分离账号：有条件的话用专门系统账号跑它，把权限与个人账号隔离
- 内网优先：远程用内网组网（Tailscale/WireGuard），别裸露公网面板
- 先确认后执行：涉及删改/发消息/登录网页，先让它给“执行计划”你确认再做

一条经验：**你允许它做的事情，一定会被它“越用越熟”。**  
所以边界要先设，别等出事了再补。

---

## “灵魂”与“记忆”：为什么它能越用越像你

Clawdbot 的长期能力通常会沉淀在工作区文件里。很多人把这些文件叫“灵魂”：

- `IDENTITY.md`：它是谁（身份与角色）
- `USER.md`：你是谁（偏好、时区、工作流）
- `SOUL.md`：它怎么说话、怎么做事（语气、边界、模板）
- `HEARTBEAT.md`：定时任务清单（每天/每小时的例行任务）
- `MEMORY.md`：长期记忆（核心知识、追踪事项）
- `TOOLS.md`：工具说明（你希望它怎么用工具）

除此之外还有 Sessions（会话历史）。会话越聊越大是正常的：  
需要干净上下文就开新会话；需要“回忆旧事”就让它去读工作区或历史会话，别指望它凭空记得。

---

## 高效用法：别把它当聊天搭子，把它当交付团队

最稳的一条规则：**你要交付物，不要聊天。**

同一句需求，写法不同，效果差一个档：

- 低效：讲讲怎么做
- 高效：给我一个可交付结果，并按格式输出

建议你养成两个习惯：

- 习惯 1：先让它输出“执行计划 + 风险 + 需要你确认的点”，确认后再执行
- 习惯 2：把高频流程固化成模板/Skill，追求复利

---

## 指令模板库（可直接复制）：从“能用”到“好用”的最快路径

信息整理（把信息变成行动）：

- “把下面链接整理成 10 条要点，每条不超过 30 字；最后输出 3 个我该做的下一步动作：……”
- “把这段对话整理成会议纪要：议题、结论、Owner、截止时间、风险点：……”

文件/知识库（把文件变成决策材料）：

- “读取 `项目A` 文件夹最新 PDF：输出 1 页摘要 + 10 个关键点 + 5 个需要我确认的问题。”
- “把下载目录今天新增文件按项目归档；执行前先给计划，执行后给目录树与变更清单。”

网页任务（把浏览器变成流水线）：

- “检查这个页面是否更新；如果更新，列出变化点，并给我下一步建议：……”
- “把这个页面的表格提取成 CSV，并按字段 A/B/C 清洗后返回。”

运维/排障（把问题变成最短路径）：

- “服务挂了：把日志与资源占用按‘最可能/次可能/低概率’三类归因，并给我最短排障路径（每一步有预期结果）。”
- “评估这台 VPS 是否过配：给出 CPU/内存/磁盘/流量结论，给降配建议与风险说明。”

写作/发布（把选题变成可发布）：

- “用公众号风格写一篇：先给提纲，再写正文，最后给 5 个标题；要求句子短、段落碎、少口水；素材：……”
- “把这篇文章改成‘给忙人看的’版本：保留信息密度，删掉空话，输出清单化结构：……”

个人助理（把碎念变成系统）：

- “把我这段语音整理成待办：每条给优先级、建议截止时间、下一步动作：……”
- “我下周要做一次分享，帮我倒排计划：准备清单、时间表、风险点、我需要提前找谁确认。”

## 真实案例清单：挑一个最痛的先跑通

下面这些案例我都尽量写成“你看完就能照抄”的形式：一句话目标 + 下单方式 + 你能拿到的交付物。

### 案例 1：语音碎念 → 待办清单（甚至能接到系统提醒）

你在 IM 里发语音（或者一段很乱的碎念），它把你要做的事整理成结构化清单：  
任务、优先级、建议截止时间、下一步动作。

下单方式：

- “把我这段语音整理成待办：每条给优先级、建议截止时间、下一步动作；最后给我一份可复制的清单：……”

你拿到的交付物：

- 一份能直接粘贴进 Notion/备忘录/Reminders 的待办清单

如果你愿意进一步配置（例如接入提醒/日历），它还能把待办写进去并设置 due date，但这一步建议你先把安全边界弄清再做。

### 案例 2：项目文件夹 → 周报/复盘材料（从“资料堆”变成“可读结论”）

这类任务最适合交给 Agent：  
你不用手动翻 PDF、翻 Markdown、翻一堆零碎文件，它直接给你“结论版本”。

下单方式：

- “读取 `项目A` 文件夹最新文档：输出 1 页摘要 + 本周进展 + 风险点 + 下周计划 + 需要我确认的问题清单。”

你拿到的交付物：

- 可直接发给团队的周报/同步材料

### 案例 3：网页任务自动化（把“刷网页”变成“可重复流程”）

Clawdbot 的一个杀手锏是它能做网页自动化（可以理解成轻量 RPA + 大模型决策）。  
你不需要每天手动打开网页确认有没有变化，它给你“变化点 + 下一步”。

下单方式：

- “盯这个页面是否更新；如果更新，列出变化点，并告诉我下一步要做什么：……”

你拿到的交付物：

- 一份变化对比 + 行动建议（而不是一句“更新了”）

### 案例 4：信息筛选 Digest（大事即时推送，小事批量汇总）

很多人最爽的用法不是“让它多干活”，而是“让它替你守夜”：  
你关心的消息源它先扫一遍，重要的立刻推你，不重要的按批次汇总。  
你也可以把它扩展成“主动联系”的通知系统：任务跑完通知你、日程提醒你、监控触发时叫醒你（但前提是你把权限与确认流程设好）。

下单方式：

- “给我做一个每天 3 次的 digest：重点 10 条，每条 1 句话总结 + 1 句我该怎么做；大事即时推送。”

你拿到的交付物：

- 可长期复用的“信息→行动”管道

### 案例 5：写作发布流水线（把“一句话想法”变成“可发布内容”）

这不是让它帮你写文章那么简单，而是让它把“写作流程”跑完：  
调研、成文、翻译、配图、排版，最后产出可发布版本；更狠一点还能把流程封装成 Skill。

下单方式：

- “我给一句话选题：你完成调研→写中文→给 5 个标题→输出公众号排版版本；最后总结成可复用模板：……”

你拿到的交付物：

- 一篇可直接发布的 Markdown/公众号排版稿

### 案例 6：运维省钱（监控一天，比你自己看半个月更有效）

这类场景非常真实：你买了台 VPS，心里一直没底——到底是不是过配？  
让 Agent 跑一天监控、看日志、做结论，往往比你“感觉”准得多。有人用这套方法把月费从约 48 美元降到约 12 美元（降配后依然满足负载）。

下单方式：

- “帮我评估这台机器是否过配：采样监控 + 看日志 + 给出降配建议与风险说明；先给执行计划我确认再跑。”

你拿到的交付物：

- 一份带证据的资源结论 + 具体降配方案

### 案例 7：当它出问题时，让它自己修（甚至自己提 PR）

Agent 最像“同事”的地方在这：  
它不是只会说“你应该怎么做”，而是能自己去定位问题、改配置/改技能，必要时把修复贡献回仓库（当然前提是你把边界设好、把确认流程走好）。

## 常见坑与避坑：这部分能帮你少走很多弯路

### 先做 5 分钟排障：按这个顺序查，效率最高

1. Gateway 是否真的在跑：先确认服务进程/守护进程状态、面板是否可访问
1. 渠道是否连上：token/扫码是否有效，是否在正确的 bot/账号里对话
1. Pairing 是否批准：第一次 DM 不回，80% 卡在这里
1. 权限是否被你自己限制死：目录沙箱/allowlist/网络策略是否过严
1. 模型鉴权是否稳定：授权掉线、额度不足、地区/网络问题都会表现成“它突然不聪明/不执行”

如果你已经拿到了明确的报错信息（日志/堆栈/提示文案），最快的方式通常是：

- 把报错原样贴给它
- 让它先输出“可能原因排序 + 最短修复路径”（每一步都写预期结果）
- 你确认后再执行改动

### 典型坑（看一次就能避开）

- 一上来就全自动：先从低风险任务开始（整理、总结、归档），再逐步放权
- 直接暴露公网端口：优先内网组网（Tailscale/WireGuard），别裸奔
- 权限给太大：最小权限原则，尤其是文件系统与命令执行
- 会话越聊越慢/越乱：开新会话，把长期规则写回工作区文件
- 模型授权不稳定：先追求稳定，必要时换鉴权方式/模型供应商

## 最后一段：从这 3 个任务开始，你就知道值不值得

如果你今天就想感受到“它真的在干活”，我建议从这 3 件事开始：

1. 发一段语音，让它整理成待办清单
1. 给它一个文件夹，让它读最新文档并输出摘要 + 行动项
1. 给它一个网页，让它监控变化并给出下一步建议

三件事跑通之后，你基本就能判断：Clawdbot 值不值得你继续“养”下去。
