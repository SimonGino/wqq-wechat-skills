# 写作风格指南

> 基于 6 篇历史文章分析总结，供 AI 生成时参考。

## 标题风格

- 主标题直接点明工具 + 价值，例：
  - "OpenCode + Oh My OpenCode 完整配置指南"
  - "Everything Claude Code 教程：少走弯路的配置路线"
  - "从 0 到生产级实战（长文收藏版）"
- 副标题常用冒号分隔，补充阅读定位
- 避免"震惊！""必看！"等标题党

## 开头模式

- **引子段**：1-3 句话抛出痛点或场景，不废话
  ```
  如果你已经在用 Claude Code，但经常遇到这些情况：
  - 输出质量忽高忽低...
  - 一换项目就乱...
  ```
- **价值承诺**：紧接着告诉读者"读完能拿到什么"
  ```
  你读完能拿到三样东西：
  1. 一套可复制的安装与启用步骤
  2. 一套可验收的最小闭环
  3. 一套"该开什么/先别开什么"的选择策略
  ```
- **资源前置**：核心链接放在开头，不让读者翻到最后才找

## 段落结构

- **短段落**：每段 2-5 行，超过就拆
- **大量使用列表**：要点用 `-` 或数字列表，避免大段文字
- **表格高密度信息**：对比、速查、配置项用表格呈现
- **代码块**：每个操作步骤配完整可运行的命令
- **分隔线**：章节间用 `---` 分隔，视觉清晰

## 章节组织

- **目录先行**：长文开头给完整目录，带序号
- **渐进式结构**：
  1. 是什么（定义/定位）
  2. 为什么（价值/收益）
  3. 怎么装（安装步骤）
  4. 怎么用（核心用法）
  5. 怎么配（配置详解）
  6. 常见坑（踩坑与排查）
  7. 附录（完整配置参考）
- **每章开头给一句话总结**：用 `> 引用格式` 或加粗

## 常用句式与口头禅

- "一句话定义：..."
- "一句话总结：..."
- "你会得到 / 你能拿到..."
- "先别折腾 / 先不要..."
- "建议先...再..."
- "最直接的变化是..."
- "这类任务最适合..."
- "如果你只想要一个结论：..."
- "按这个顺序：..."

## 过渡词

- 递进："除此之外"、"更具体一点"、"进一步"
- 转折："但"、"不过"、"问题是"
- 总结："所以"、"一句话"、"核心是"
- 条件："如果你..."、"当你..."

## 代码示例风格

- **先给完整可运行命令，再解释**
- **bash 命令带注释**：用 `#` 说明每步做什么
- **配置文件用 jsonc**：带 `//` 注释解释每个字段
- **给出验证方法**：每个步骤后告诉读者怎么确认成功
- **错误处理**：常见报错直接给解决方案

## 信息密度

- **干货优先**：不铺垫，直接给答案
- **要点清单化**：能用列表就不用段落
- **表格速查**：命令、配置、对比都用表格
- **可复制性**：读者能直接复制粘贴使用

## 视觉元素

- **截图配文字说明**：截图后紧跟一句话解释在哪/怎么操作
- **流程用步骤号**：1. 2. 3. 清晰
- **重点加粗**：关键术语、命令、警告用 `**加粗**`
- **引用块**：用于提示、警告、总结
  ```
  > **注意**：如果 bunx 不行就换 npx。
  ```

## 禁忌

- 不用"亲爱的读者"、"小伙伴们"
- 不用过多 emoji（偶尔用 ✅ ⚠️ 等功能性图标可以）
- 不用"废话开场"（如"今天我们来聊聊..."）
- 不用长难句，句子超过 2 行就拆
- 不用模糊表达（"可能"、"大概"、"应该"），给确定结论
- 不做无意义的"背景介绍"，直接进主题
- 不重复啰嗦，一个点说一次

## 结尾模式

- **行动建议**：告诉读者下一步做什么
  ```
  如果你今天就想感受到"它真的在干活"，我建议从这 3 件事开始：
  1. ...
  2. ...
  3. ...
  ```
- **资源链接汇总**：表格形式给出相关链接
- 不用"感谢阅读"、"欢迎点赞"等套话

## 受众假设

- 读者是**有一定技术基础的开发者**
- 读者**时间宝贵**，要快速获取可执行信息
- 读者**会动手操作**，所以命令必须可运行
- 读者**遇到问题会回来查**，所以结构要方便跳转

## 语言风格

- **口语化但不随意**：像在和同事讲解，不是写论文
- **直接**：不绕弯，不客套
- **自信**：给建议时用肯定语气，不犹豫
- **实用主义**：只讲有用的，不讲"正确但没用的"
